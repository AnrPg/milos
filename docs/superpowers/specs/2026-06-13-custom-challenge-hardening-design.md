# Custom Challenge Hardening — Design Spec
Date: 2026-06-13
Status: Approved (rev 2 — rich rules system added)

## Overview

Harden the `custom` criteria type for seasonal challenges end-to-end:
fix a landing-page target computation bug, replace the single
`increment_per_completion` integer with a rich multi-condition rules engine,
expose progress semantics to athletes with ephemeral per-completion breakdowns,
add a per-challenge opt-in Hall of Fame leaderboard (expandable inline on the
landing page), and improve the admin form and participant table.

---

## 1. Data Model Changes

### 1a. `criteria_value` for `custom` type — rich rules format

The `custom` criteria type now uses a `rules` array instead of (or in
addition to, for backward compat) `increment_per_completion`.

**New format:**
```json
{
  "count": 100,
  "rules": [
    { "condition": "workout_type", "type": "crossfit", "points": 3, "label": "for a CrossFit workout" },
    { "condition": "workout_type", "type": "strength",  "points": 2, "label": "for a Strength workout" },
    { "condition": "scale_level",  "slug": "rx",        "points": 2, "label": "for RX performance" },
    { "condition": "pr_beaten",    "points": 5,         "label": "for beating a PR" },
    { "condition": "weekly_consistency", "threshold": 50, "points": 3, "label": "for weekly consistency" },
    { "condition": "rare_workout_type",  "threshold_pct": 10, "points": 4, "label": "for exploring a new type" },
    { "condition": "team_workout_streak", "min_count": 2,  "points": 3, "label": "for team participation" }
  ]
}
```

**Backward-compat format** (existing challenges with `increment_per_completion`):
```json
{ "count": 20, "increment_per_completion": 5, "increment_label": "for any completion" }
```

Detection: if `criteria_value` contains `"rules"` key → use rules engine.
Otherwise fall back to `increment_per_completion` (old behaviour, unchanged).

**Rule condition types:**

| condition | required params | fires when |
|---|---|---|
| `workout_type` | `type` (string), `points` | current workout type matches `type` |
| `scale_level` | `slug` (string), `points` | `execution.scale_level_slug` matches `slug` |
| `pr_beaten` | `points` | `pr_count > 0` |
| `weekly_consistency` | `threshold` (0–100 int), `points` | `consistency_score * 100 >= threshold` |
| `rare_workout_type` | `threshold_pct` (0–100 int, default 10), `points` | `(prev_type_count / total_prev) * 100 < threshold_pct` OR first-ever workout |
| `team_workout_streak` | `min_count` (int ≥ 1), `points` | team workouts in challenge period ≥ `min_count` |

All matching rules are **additive** (points summed). A single completion can
match multiple rules.

Each rule has an optional `label` string (max 160 chars). Default labels are
generated by `ChallengeRule.default_label/1` if absent.

**Validation** in `ChallengeCriteria.normalize_value("custom", count, value)`:
- `count` required, positive integer.
- If `rules` present: non-empty array; each rule validated against its
  condition schema (required params, `points` ≥ 1).
- If no `rules` and no `increment_per_completion`: validation error.
- `increment_per_completion` (legacy) still accepted unchanged.

### 1b. `user_challenge_progress` — add `last_increment_event`

New nullable JSONB column `last_increment_event`.

**Simple format** (legacy/single increment):
```json
{ "total_points": 5, "events": [{ "points": 5, "label": "for any completion" }] }
```

**Rich format** (multi-rule):
```json
{
  "total_points": 10,
  "events": [
    { "points": 3, "label": "for a CrossFit workout" },
    { "points": 5, "label": "for beating a PR" },
    { "points": 2, "label": "for RX performance" }
  ]
}
```

Set on every `upsert_user_challenge_progress` call with `increment > 0`.
Persists across page reloads (overwritten only by the next progress step).

Migration: `add :last_increment_event, :map, default: nil`

### 1c. `challenge_leaderboard_opt_ins` — new table

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | |
| `user_id` | uuid FK → users | |
| `challenge_id` | uuid FK → seasonal_challenges ON DELETE CASCADE | |
| `inserted_at` | utc_datetime_usec | |

Unique index: `(user_id, challenge_id)`. Default: **opted-out** (no row = not in Hall of Fame).

## 2. Domain Layer

### `ChallengeCriteria`

- `normalize_value("custom", count, value)`: validate both formats (rules array and legacy).
- New helper: `has_rules?(cv)` — `true` if `criteria_value["rules"]` is non-empty list.
- Legacy helpers preserved: `increment_per_completion/1`, `increment_label/1`.

### New: `ChallengeRule` module

```elixir
defmodule MilosTraining.Gamification.Domain.ChallengeRule do
  # evaluate one rule against completion_facts
  # returns 0 if condition not met, `rule["points"]` if met
  @spec evaluate(rule :: map(), facts :: map()) :: non_neg_integer()

  # human-readable label for this rule's point event
  @spec label(rule :: map()) :: String.t()

  # default label per condition type (used when rule has no "label" key)
  @spec default_label(rule :: map()) :: String.t()
end
```

**`completion_facts` map** (extended from current `%{workout_type, pr_count}`):

```elixir
%{
  workout_type:                   string | nil,
  pr_count:                       non_neg_integer(),
  scale_level_slug:               string | nil,          # from execution.scale_level_slug
  consistency_score:              float(),               # from streaks.consistency_score (0.0–1.0)
  prev_type_count:                non_neg_integer(),     # times this type done BEFORE current completion
  total_prev_completions:         non_neg_integer(),     # total completions BEFORE current
  team_workout_count_fn:          (challenge -> integer) # fn computed lazily per challenge
}
```

`team_workout_count_fn` is a closure over `completed_executions` and `workout_lookup`,
evaluated per challenge (because the challenge period differs).

### `ChallengeProgress`

- `increment/2` for `custom` type gains a branch:
  - `if has_rules?(cv)`: call `ChallengeRule.evaluate/2` for each rule, sum results, return total.
  - else: use legacy `ChallengeCriteria.increment_per_completion/1`.
- `increment/2` signature unchanged; `completion_facts` map is extended (new keys are ignored by non-custom types).
- New `increment_events/2` returns the list of `{points, label}` for matched rules (used for `last_increment_event`).

### New: `ChallengeLeaderboard`

```elixir
defmodule MilosTraining.Gamification.Domain.ChallengeLeaderboard do
  @spec rank(participants :: list(map())) :: list(map())
  # adds :rank field, handles ties (equal progress → same rank)
end
```

## 3. Application Layer

### `RecordWorkoutCompletion`

**`call/1` — build extended `completion_facts`** before entering the transaction:

```elixir
type_counts = workout_type_counts(completed_executions, workout_lookup)
current_type = workout_lookup[execution.master_workout_id][:type]
total_prev = length(completed_executions) - 1
prev_type_count = max(0, Map.get(type_counts, current_type, 0) - 1)

completion_facts = %{
  workout_type: current_type,
  pr_count: length(pr_scores),
  scale_level_slug: execution.scale_level_slug,
  consistency_score: streaks.consistency_score,
  prev_type_count: prev_type_count,
  total_prev_completions: total_prev,
  team_workout_count_fn: fn challenge ->
    team_workout_count_for_challenge(challenge, completed_executions, workout_lookup)
  end
}
```

Pass `completion_facts` to `persist_challenge_progress/5` (replaces separate `pr_count` arg).

**`persist_challenge_progress`** — changes:
- For each challenge, call `completion_facts.team_workout_count_fn.(challenge)` to populate the
  lazy field before advancing.
- Check opt-in status: if opted-out → `next_progress = min(update.progress, target)`.
- Build `last_increment_event` from `ChallengeProgress.increment_events/2`.
- Pass `last_increment_event` to `upsert_user_challenge_progress`.

**Return value extended:**
```elixir
{:ok, %{
  challenge_completions: [...],    # existing: challenges just completed
  challenge_increments: [          # new: every non-zero progress event this completion
    %{challenge_id: id, title: title, total_points: integer,
      events: [%{points: integer, label: string}]}
  ]
}}
```

### `GetLandingPage`

**Bug fix**: replace raw `Map.get(challenge.criteria_value, "count")` with
`ChallengeProgress.target(challenge)`.

**New payload fields per challenge:**
```elixir
%{
  # existing
  id, title, description, badge_key, badge_label,
  criteria_type, target, progress, completed_at, completed,
  starts_at, ends_at,
  # new
  increment_per_completion: integer | nil,      # legacy format only
  has_rules: boolean,                           # true if using rich rules engine
  completions_remaining: integer,               # max(0, ceil((target-progress)/effective_increment))
  is_opted_in: boolean,
  last_progress_event: %{total_points: integer, events: [...]} | nil
}
```

`completions_remaining` computation:
- Rules format: `max(0, ceil((target - progress) / average_rule_value))` — average of all rules'
  max possible points per completion.
- Legacy format: `max(0, ceil((target - progress) / increment_per_completion))`.
- Non-custom types: `max(0, target - progress)`.

### `CompleteWorkout`

`process_completion/1` broadcasts after success:
```elixir
BroadcastUserSync.for_user(user_id, ["landing"],
  reason: "challenge_progress_advanced",
  payload: %{increments: challenge_increments}
)
```

### New services: `OptInChallengeLeaderboard`, `OptOutChallengeLeaderboard`, `GetChallengeLeaderboard`

Unchanged from rev 1.

### `GetAdminChallenge`

For custom challenges, each participant row gains:

```elixir
completions_done: integer | nil,      # floor(progress / effective_increment)

```

## 6. Frontend Changes

### `apps/web/src/api/challenges.ts`

- New types: `ChallengeRule`, `RichCriteriaValue`, `ChallengeLeaderboardEntry`.
- New functions: `optInChallenge`, `optOutChallenge`, `fetchChallengeLeaderboard`.

### `RealtimeSyncBridge`

`challenge_progress_advanced` reason → invalidates `["landing"]` query only.
No separate local state needed; `last_progress_event` comes from the DB via refetch.

### `admin-challenges.tsx` — Rules Builder

When `criteriaType === "custom"`, the form expands to show the **Rules Builder**
instead of the simple `incrementPerCompletion` + `incrementLabel` inputs:

```
┌─────────────────────────────────────────────────────────────┐
│ Target points   [100          ]                             │
│                                                             │
│ Rules                                         [+ Add rule] │
├──────────────────────────────────────┬──────┬──────────────┤
│ Condition                            │ Pts  │              │
├──────────────────────────────────────┼──────┼──────────────┤
│ [Workout type ▼] [crossfit ▼]        │ [3]  │ [×]          │
│ label: "for a CrossFit workout"      │      │              │
├──────────────────────────────────────┼──────┼──────────────┤
│ [PR Beaten ▼]                        │ [5]  │ [×]          │
│ label: "for beating a PR"            │      │              │
├──────────────────────────────────────┼──────┼──────────────┤
│ [Weekly consistency ▼] ≥ [50]%       │ [3]  │ [×]          │
│ label: "for weekly consistency"      │      │              │
└──────────────────────────────────────┴──────┴──────────────┘
  Preview: "Reach 100 pts · up to 11 pts per workout"
```

Per-condition sub-fields:
- `workout_type`: type dropdown (all 6 training types)
- `scale_level`: slug dropdown (from existing scale levels)
- `pr_beaten`: no sub-params
- `weekly_consistency`: threshold % input (default 50)
- `rare_workout_type`: threshold_pct input (default 10)
- `team_workout_streak`: min_count input (default 2)

Each row has an optional collapsible "Label" input pre-filled with
`ChallengeRule.default_label` equivalent.

**Form serialization**: `payloadFromForm` builds `criteria_value.rules` array.
**Form hydration**: `hydrateForm` reads `criteria_value.rules` and reconstructs
rule rows.

**`criteriaSummary`** for custom:
- No rules: `"Reach N pts (+M per completion)"` (legacy)
- With rules: `"Reach N pts · up to X pts per workout"` where X = sum of all rule points

**Input guards**: `min={1}` on all `points` inputs; `min={1}` on `min_count`.

### `admin-challenges.tsx` — Participant table

For custom challenges, add columns: **Done** and **Remaining** (completions).

Progress bar fill is **capped visually at 100%** even if `progress > target`.
Raw numbers (`25/20 pts`) are shown above the bar when opted-in and past target.

## 7. Realtime Flow (unchanged from rev 1)

```
Athlete completes workout
  → RecordWorkoutCompletion builds extended completion_facts
  → persist_challenge_progress: evaluates all rules per challenge
  → upsert_user_challenge_progress (persists last_increment_event, respects cap)
  → returns challenge_increments
  → CompleteWorkout.process_completion
  → BroadcastUserSync(["landing"], reason: "challenge_progress_advanced")
  → RealtimeSyncBridge invalidates ["landing"] React Query cache
  → LandingPage refetch → landing payload has updated progress + last_progress_event
  → LandingPage re-renders with new points breakdown text
```

---

## 8. Out of Scope

- Dynamic per-completion reasons that differ by context (e.g., "for this specific exercise") — rules labels are static per rule.
- Hall of Fame pagination beyond top 50.
- Admin management of opt-in lists.
- Retroactive points recalculation when challenge rules are edited after participants have progress.
